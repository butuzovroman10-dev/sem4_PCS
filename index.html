📚 Что такое полиморфизм?

Полиморфизм (от греч. πολύς — много + μορφή — форма) — это способность объектов разных классов реагировать на один и тот же вызов метода по-разному, в зависимости от их реального типа.

🎯 Ключевая идея: Один интерфейс — множество реализаций. Клиентский код работает с базовым интерфолсом и не знает о конкретных типах объектов.

✅ Преимущества полиморфизма

  • Унификация кода через общий интерфейс
  • Легкое добавление новых типов
  • Снижение связанности компонентов
  • Реализация принципа открытости/закрытости (OCP)

🎯 Где применяется

  • Игровые движки (разные типы юнитов)
  • Графические редакторы (фигуры)
  • Системы плагинов
  • UI-фреймворки (виджеты)

🏗️ Иерархия классов нашей IT-компании

🔷 Абстрактный интерфейс IWorkable: задаёт контракт — каждый работник должен уметь выполнять работу (work()) и сообщать свою эффективность (getEfficiency()). Создать объект IWorkable напрямую невозможно — только через конкретные классы.

💻 Реализация интерфейса и классов

Абстрактный интерфейс IWorkable

 
class IWorkable {
public:
    virtual void work() const = 0;           // чисто виртуальный метод
    virtual double getEfficiency() const = 0; // чисто виртуальный метод
    virtual ~IWorkable() = default;              // виртуальный деструктор
};

Базовый класс Employee

class Employee : public IWorkable {
protected:
    std::string name;
    int id;
    double baseSalary;
public:
    Employee(const std::string& n, int i, double salary);
    // Оставляем методы чисто виртуальными → Employee тоже абстрактный
    void work() const override = 0;
    double getEfficiency() const override = 0;
    virtual void displayInfo() const = 0;
};

Производный класс Developer (с override)

class Developer : public Employee {
private:
    std::string programmingLanguage;
    int linesOfCodePerDay;
public:
    // Переопределение с ключевым словом override
    void work() const override {
        std::cout << "Developer " << name << " writes " 
                  << linesOfCodePerDay << " lines of " 
                  << programmingLanguage << std::endl;
    }
    
    double getEfficiency() const override {
        return std::min(1.0, linesOfCodePerDay / 500.0);
    }
};

Производный класс Manager

class Manager : public Employee {

🎭 Полиморфная функция analyzeWorker()

// Функция принимает ссылку на интерфейс IWorkable
void analyzeWorker(const IWorkable& worker) {
    std::cout << "=== Анализ работника ===" << std::endl;
    
    // Полиморфный вызов — будет вызвана нужная версия work()
    worker.work();
    
    // Полиморфный вызов getEfficiency()
    double eff = worker.getEfficiency();
    std::cout << "Эффективность: " << eff * 100 << "%" << std::endl;
    
    // Логика не зависит от конкретного типа!
    if (eff > 0.8) {
        std::cout << "Статус: Отличный сотрудник" << std::endl;
    } else if (eff > 0.5) {
        std::cout << "Статус: Удовлетворительно" << std::endl;
    } else {
        std::cout << "Статус: Требуется улучшение" << std::endl;
    }
}
🌟 Важно: Функция analyzeWorker ничего не знает о классах Developer, Manager, Tester. Она работает только с интерфейсом IWorkable. Тем не менее, при передаче объектов разных типов вызываются их собственные реализации методов. Это и есть полиморфизм.

🔍 RTTI: typeid и dynamic_cast

// Демонстрация получения информации о типе во время выполнения
void demonstrateTypeChecking() {
    for (const auto& emp : employees) {
        // typeid - получает информацию о реальном типе объекта
        std::cout << "Тип объекта: " << typeid(*emp).name() << std::endl;
        
        // dynamic_cast - безопасное приведение к производному типу
        if (auto dev = dynamic_cast<Developer*>(emp.get())) {
            std::cout << "Это разработчик. Язык: " << dev->getLanguage() << std::endl;
        }
    }
}

⚠️ Важное предупреждение о dynamic_cast и typeid:
  • Когда может потребоваться: Для доступа к специфическим методам, которых нет в базовом интерфейсе.
  • Почему это часто антипаттерн: Постоянные проверки типов нарушают принцип подстановки Лисков (LSP). Если код знает о конкретных типах, он теряет гибкость — при добавлении нового класса придётся переписывать все проверки.
  • Правильное решение: Выносить специфическое поведение в интерфейс или использовать паттерн "Посетитель" (Visitor).

✅ Корректное освобождение при удалении контейнера
✅ Безопасность исключений

🔧 Виртуальный деструктор

class IWorkable {
public:
    virtual ~IWorkable() = default;
};

✅ При удалении через указатель на базовый класс
✅ Вызываются все деструкторы (базовый → производный)
✅ Нет утечек памяти

💡 Как это работает: Когда std::unique_ptr<IWorkable> уничтожается, он вызывает деструктор IWorkable. Благодаря виртуальности, вызывается правильная цепочка деструкторов: сначала ~Developer(), затем ~Employee(), потом ~IWorkable().

  • Упрощается тестирование (можно подставлять моки)

📖 Принцип подстановки Лисков (LSP): Функции, использующие базовый тип, должны иметь возможность работать с производными типами, ничего не зная о них. Наша функция analyzeWorker следует этому принципу — она корректно работает с любым наследником IWorkable, и для добавления Designer или DevOps не потребуется её изменять.
